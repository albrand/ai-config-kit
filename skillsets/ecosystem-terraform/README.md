# Ecosystem Terraform Skillset

Use this skillset when a user wants an AI to bootstrap or harden the operating ecosystem around a project, not just write code.

For a user-facing invitation, command chooser, and copy-paste prompt samples, read `ECOSYSTEM_TERRAFORM_GUIDE.md`.

The workflows are Claude Code friendly and also have Codex-compatible skill mirrors:

- `/roadmap-terraform`: turn business material, docs, designs, current boards, and stakeholder intent into a durable roadmap, project operating model, and PR-sized ticket ecosystem.
- `/tech-terraform`: turn a chosen or discovered technical stack into repo, environment, CI/CD, PR review, infrastructure, quality-gate, and developer-workstream bootstrap artifacts.
- `/assess-then-harden`: read an existing project deeply, generate or refresh knowledge docs, identify missing requirements and tickets, and produce a hardening plan tied to external source-of-truth material.

The word `terraform` means "create or reconcile the project operating system from evidence." It does not mean Terraform IaC unless the user explicitly asks for HashiCorp/OpenTofu infrastructure code.

## Runtime Model

These are AI-runbook workflows. They are executable as Claude Code slash commands, but they should not depend on custom scripts.

- Use normal AI capabilities: read files, inspect docs, search repositories, inspect connected boards through MCP when available, inspect designs, reason over evidence, ask questions, and write/update artifacts only when asked.
- Read-only discovery commands such as `rg`, `find`, `git status`, `git diff`, and tool-native search are allowed when they reduce uncertainty.
- Do not connect cloud providers, mutate boards, create repositories, change secrets, provision resources, or spawn sub-agent swarms until the user has approved that action and the required capability is verified.
- Ask targeted questions whenever missing information would materially change scope, sequencing, governance, or generated artifacts. Do not ask for information that can be discovered safely from provided sources.
- Prefer updating existing projects, boards, docs, labels, milestones, and roadmaps over creating duplicates.

## Single-Prompt Operator Mode

Many users will invoke these commands with a broad prompt and expect the agent to "make the project system appear." In that mode, the agent must not pretend the prompt was complete.

The agent should:

- Translate the broad prompt into a concrete operating model, source list, capability gate, and artifact plan.
- Ask only the questions that change the next irreversible or expensive step.
- Recommend a default answer for each question when the evidence supports one.
- Offer paste-ready artifacts when live MCP/CLI writes are unavailable or unapproved.
- Keep non-technical users oriented with business-language summaries while preserving engineering traceability.
- Stop before mutating external systems, launching swarms, handling secrets, or choosing architecture/cloud directions without approval.

## Imported Practice Patterns

This skillset adapts public engineering patterns without requiring any external dependency:

- High-signal PR review: preflight stop checks, independent reviewers, issue validation before comments, and no duplicate comments.
- Grilling before commitment: one decision at a time, with recommended answers and code/doc discovery before asking.
- Domain language and ADR capture: create glossary and ADR entries lazily only when decisions crystallize.
- Vertical-slice ticketing: one ticket should produce one independently verifiable PR.
- Triage state discipline: distinguish needs-info, ready-for-agent, ready-for-human, and wontfix instead of dumping vague tickets onto a board.
- TDD and diagnosis loops: keep feedback loops explicit for implementation tickets generated by the roadmap or tech workflows.
- Architecture deepening: prefer deeper modules, narrower interfaces, and testable seams when hardening legacy projects.
- Handoff hygiene: create concise resume packets when work crosses agents, sessions, or tools.

## Install

Claude Code:

0. Read `ECOSYSTEM_TERRAFORM_GUIDE.md` for usage examples and prompt samples.
1. Copy `claude/commands/*.md` to `~/.claude/commands/` or the target repo `.claude/commands/`.
2. Copy or keep `references/ecosystem-output-contract.md` under the same framework folder and point the commands to it if the installed path differs.
3. Run `/roadmap-terraform <prompt>`, `/tech-terraform <prompt>`, or `/assess-then-harden <prompt>`.

Codex:

0. Read `ECOSYSTEM_TERRAFORM_GUIDE.md` for usage examples and prompt samples.
1. Copy each `codex/<skill-name>/SKILL.md` to `<CODEX_HOME>/skills/<skill-name>/SKILL.md`.
2. Copy `references/ecosystem-output-contract.md` into each skill's `references/` folder or provide the framework path in the prompt.

Generic AI:

1. Paste `ECOSYSTEM_TERRAFORM_GUIDE.md`.
2. Paste this README and `references/ecosystem-output-contract.md`.
3. Paste the relevant command or `SKILL.md`.
4. Provide the project prompt plus docs, board, repo, design, cloud, and QA-source access.

## Completion Bar

Each workflow is complete only when it reports:

- Sources inspected and sources unavailable.
- Questions asked, answered, deferred, or safely defaulted.
- Capabilities verified, blocked, or not used.
- Artifacts created, updated, left unchanged, or proposed only.
- Ticket and roadmap changes tied to source evidence.
- QA and quality gates tied to business logic, docs, tickets, and external designs.
- Approval gates still required before any sub-agent swarm, board mutation, cloud mutation, repository creation, or secret handling.
